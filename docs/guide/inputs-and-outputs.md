# Requirements, Supported Inputs, and Outputs

This page answers the practical questions most users ask before adopting CortexMark:

1. **What do I need to install?**
2. **Which tools are optional, and when are they actually used?**
3. **What kinds of files can CortexMark process?**
4. **What artifacts will be produced?**

## Runtime requirements

### Minimum runtime

For the main pipeline, the minimum supported runtime is:

- **Python 3.11 or newer**
- A normal `pip`-based installation of **`cortexmark`**
- A folder or file containing **PDF input**

The default installation is lightweight and CPU-only:

```bash
pip install cortexmark
```

This installs the default runtime dependencies declared in `pyproject.toml`:

- `markitdown[pdf]`
- `PyYAML`

That is enough to run the pipeline with the **`markitdown`** engine.

### Optional runtime components

Some capabilities require additional packages or host tools.

| Component | How to install | When it is used | Required? |
|---|---|---|---|
| **Docling engine** | `pip install "cortexmark[docling]"` | Used by `--engine docling` and `--engine dual` for layout-aware conversion, formulas, and structure recovery | Optional |
| **GPU-oriented install path** | `pip install "cortexmark[gpu]"` | Same features as Docling, but intended for CUDA-capable systems | Optional |
| **PyTorch CPU build** | Install before `cortexmark[docling]` | Recommended on CPU-only machines to avoid accidental CUDA downloads | Optional but recommended for Docling on CPU |
| **Poppler** | `poppler-utils` / `poppler` | Helpful for some PDF backends and OCR-adjacent workflows | Optional |
| **Tesseract OCR** | `tesseract-ocr` / `tesseract` | Used only when OCR-style workflows are needed or when your PDFs are image-heavy/scanned | Optional |
| **Docker** | Docker / Docker Compose | Containerized execution | Optional |
| **VS Code** | VS Code 1.92+ | Required only for the extension experience | Optional |

> CortexMark does **not** require an LLM, API key, or cloud service for the core pipeline.

## Which tools are actually active?

The easiest way to reason about installation is to map tools to runtime behavior.

| Installed tool / package | What CortexMark does with it |
|---|---|
| `cortexmark` | Provides the Python package and the `cortexmark` CLI |
| `markitdown[pdf]` | Performs PDF → Markdown extraction in `markitdown` mode; also acts as the text-recovery engine in `dual` mode |
| `docling` | Performs layout-aware PDF parsing in `docling` mode; also acts as the structural backbone in `dual` mode |
| `PyYAML` | Reads pipeline configuration from `configs/pipeline.yaml` |
| PyTorch | Supports Docling runtime; CPU or GPU choice depends on how you install it |
| Poppler / Tesseract | Only used when the host environment and your document profile require them |
| MkDocs | Only needed if you want to build the project documentation locally |
| `@vscode/vsce`, `npm`, `tsc` | Only needed for extension development or packaging, not for end users installing from the marketplace |

## Supported input types

### Full pipeline input

The **full conversion pipeline starts from PDF files only**.

Supported at the conversion entrypoint:

- a single **`.pdf`** file
- a directory tree containing one or more **`.pdf`** files

Examples:

```bash
cortexmark --input path/to/paper.pdf
cortexmark --input path/to/folder-of-pdfs
```

### Downstream module input

After conversion, most analysis/export modules operate on **Markdown** trees generated by CortexMark.

| Module family | Expected input |
|---|---|
| `clean`, `chunk`, `metadata`, `citations`, `doc_type`, `topics`, `figures`, `rag_export`, `semantic_chunk`, `cross_ref`, `algorithm_extract`, `notation_glossary`, `formula_validate`, `citation_context`, `scientific_qa`, `ghpages`, `multi_format` | **`.md`** files or directories containing `.md` files |
| `diff` | two Markdown trees |
| `render_templates` | Markdown outputs plus optional outline/syllabus text files |

### Auxiliary source files

Some optional workflows also read helper files such as:

- `00_meta/outline.md`
- `00_meta/source_outline.md`
- `00_meta/syllabus.md`
- `00_meta/course_syllabus.md`
- outline/syllabus files matching `*outline*.md`, `*outline*.txt`, `*syllabus*.md`, `*syllabus*.txt`

These are not primary inputs; they enrich the render/template stage.

## Document profiles that work well

CortexMark is designed primarily for:

- academic papers
- lecture notes
- textbooks and book chapters
- reports and theses
- mathematical / theorem-heavy documents
- code / algorithm-heavy technical PDFs
- scanned or noisy PDFs where a fallback text-recovery path is useful

## Output structure

By default, outputs are written under `outputs/`.

| Output path | Produced by | What it contains |
|---|---|---|
| `outputs/raw_md/` | `convert` | First-pass Markdown converted from PDFs |
| `outputs/cleaned_md/` | `clean` | Normalized Markdown with repeated headers/footers and line-wrap artifacts reduced |
| `outputs/chunks/` | `chunk` | Section-based chunk files such as `chunk_001_Introduction.md` |
| `outputs/semantic_chunks/` | `analyze → semantic_chunk` | Scientific-object-aware chunks and metadata |
| `outputs/quality/` | `validate` and analysis modules | JSON / Markdown quality reports |
| `outputs/templates/` or render-specific folders | `render_templates` | Deterministic source profile / section templates |

### Common quality and analysis artifacts

Typical files include:

- `crossref_report.json`
- `algorithm_report.json`
- `notation_report.json`
- `formula_validation.json`
- `scientific_qa.json`
- `citation_context.json`
- RAG export JSON / JSONL artifacts
- multi-format exports such as HTML, plain text, or YAML front-matter views
- GitHub Pages-compatible HTML exports

### Session-scoped output layout

If you run with `--session-name`, CortexMark isolates data under:

```text
sessions/<session-name>/
├── data/raw/
└── outputs/
    ├── raw_md/
    ├── cleaned_md/
    ├── chunks/
    ├── semantic_chunks/
    └── quality/
```

This is the layout used by the VS Code extension as well.

## Practical recommendation

If you are new to the project:

1. Start with `pip install cortexmark`
2. Process one or two PDFs with `--engine markitdown`
3. Move to `cortexmark[docling]` only when you need stronger layout recovery or the `dual` engine
4. Add OCR/system tools only if your PDFs are image-heavy or your environment diagnostics indicate they are needed
